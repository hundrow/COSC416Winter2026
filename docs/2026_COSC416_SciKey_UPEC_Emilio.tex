\section{Introduction}

HAL is an open access archive where researchers can record metadata and full text from their scientific publications, including books, articles, and, in some cases, research datasets or software code repositories. Currently, it contains over 3,760,000 references and nearly 1,400,000 scientific documents, providing a rich resource for bibliometric and content analysis.

The goal of this project is to determine and automate the extraction of keywords that describe the content of publications, based primarily on their abstracts and titles. These keywords are then linked to Wikidata or BNF (Bibliothèque Nationale de France) identifiers, allowing users to characterize publications and explore relationships between authors and their work. 

Our team focuses primarily on the database aspect of this project, including the design, optimization, and querying of tables to efficiently store and analyze the extracted data. In particular, the project aims to:
\begin{enumerate}
    \item Implement efficient storage and retrieval mechanisms for large-scale publication datasets.
    \item Automate the extraction and external linking of descriptive keywords.
    \item Support comprehensive bibliometric and content analysis, including author collaborations and research domain trends.
\end{enumerate}

By achieving these objectives, the project contributes both practical improvements to database performance and novel approaches to linking bibliometric data with external knowledge bases.

\section{Methodology and Project Design}

Our team proposes to provide comprehensive Database Administration and Performance
Tuning services for the SciKey project, focusing on a scalable Oracle 19c database
architecture for HAL metadata and UPEC publications. We will apply the COSC 416
top-down tuning methodology (Design → SQL → Instance) to deliver a robust, secure, and
high-performance database system. This database could be used by the UPEC librarians or prospective authors to find similar papers by keywords or abstract content.

\subsection{Database Schema and Core Entities}

The database schema follows a star schema data warehouse design, optimized for analytical queries and indexing. Centered around the core entity RESEARCH\_ARTICLES. The ERD (Figure \ref{fig:HAL_Schema_Design}) shows the following core tables:
\begin{enumerate}
    \item RESEARCH\_ARTICLES: Stores metadata such as unique article identifiers (DOCID), titles (TITLE\_1, TITLE\_2), labels, and publication year (PUBLISH\_YEAR).
    \item ARTICLE\_AUTHORS: Contains authorship information linked to articles via DOCID, supporting multi-author scenarios.
    \item ARTICLE\_DOMAINS\_EN and ARTICLE\_DOMAINS\_FR: Provide domain categorizations for articles in English and French respectively.
    \item ARTICLE\_ABSTRACTS: Holds large textual abstracts connected to each article.
    \item ARTICLE\_KEYWORDS: Stores keywords linked to articles, supporting efficient keyword-based search.
    \item ARTICLE\_COLLECTIONS: Groups articles into collections using collection codes.
    \item ARTICLE\_DOMAINS\_CODE: Manages domain codes linked to articles for classification purposes.
\end{enumerate}

\subsection{Performance Gate Service-Level Objectives}
For transition to operations, the team defined a pass/fail Performance Gate. These SLOs are measured during a 15-minute steady-state workload interval and validated using Linux host telemetry, SQL Developer query timing, and OEM/AWR evidence.
